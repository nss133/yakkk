# 감독규정·법령 연결 (Phase 3) 설계

> 작성: 2026-07-10 · 상태: 설계 승인 대기
> 선행: Phase 1(타사 비교)·Phase 2(표준약관 준수성)·랭킹정밀도 개선(제목가중+부정어). 엔진 공용.

## 1. 배경·목적

약관 초안 심사에서 **감독규정·법령**은 최상위 규범 근거다. Phase 3은 보험업감독규정·보험업법(시행령·시행규칙)을
DB에 적재하고, 초안 조문에 **관련 감독규정·법령 조문을 자동으로 연결**해 준수성 심사의 규범 축을 완성한다.

## 2. 핵심 난점과 해법

조문↔규정은 조문↔조문과 성격이 다르다 — 규정은 "회사는 ~하여야 한다"(규범문), 약관은 이를 이행하는 문장이라
어휘가 달라 **순수 본문 유사도가 약하다**(실측: 약관 조문의 법령 명시 인용은 ~1%뿐).

**해법**: 방금 도입한 **제목 코사인 가중**을 지렛대로 쓴다. 감독규정 조문 제목은 깔끔한 주제 라벨
(`제7-45조(해약환급금)`, `제5-6조(보험금 지급)`)이라, 약관 조문을 규정 **제목**에 매칭하면 어휘 차이를 넘어
주제로 연결된다. 즉 규정을 REG 코퍼스로 적재하고 `db_similar(doc_type='REG')`로 조회하는 것이 곧 자동 매핑이다.

**단, 매핑 품질은 적재 후 실측으로 검증**하고, 약하면 주제사전(topic→규정) 보강 층을 추가한다(단계 설계).

## 3. 범위

**IN**
- 규정 3~4종 적재(doc_type='REG'): 보험업감독규정, 보험업법, 보험업법 시행령, 보험업법 시행규칙.
- 유사조문 결과에 "📖 관련 감독규정·법령" 섹션 추가(자동 매핑, `db_similar(doc_type='REG')`).
- 매핑 품질 실측 게이트 → (약하면) 주제사전 보강.

**OUT (후속)**
- 인용 링크 해소(약관이 명시 인용한 법령 펼치기) — 사용자 미선택.
- LLM 기반 의미 매핑(폐쇄망 제약 유지).
- 준수 여부 판정(사람의 판단 영역).

## 4. 데이터 모델·적재 (Phase 2 패턴 재사용)

- `catalog/standards.json` 확장 또는 신규 `catalog/regulations.json` 매니페스트:
  각 규정 = `{key, name, member_cd, version_label, source_path, source_type}`.
  - member_cd: 감독규정 `REG_GD`, 보험업법 `REG_LAW`, 시행령 `REG_LAWE`, 시행규칙 `REG_LAWR`(발행처명이 UI 라벨).
  - version_label: 시행일(예: `250218 시행`).
- 소스(로컬 vault, `08. 자원·참고자료/법령자료/법령(전문)/`):
  `보험업감독규정(전문).md`, `보험업법(전문).md`, `보험업법 시행령(전문).md`, `보험업법 시행규칙(전문) 250218 시행.md`.
- 적재: 기존 `ingest_standards.py`를 일반화하거나 `ingest_regulations.py` 추가.
  - ⚠️ 규정 조문번호는 **`제N-M조`**(감독규정) 및 `제N조`(법령) 혼재 → `clause_split`의 `제N조` 정규식이
    `제7-45조` 형식을 잡는지 확인·보강 필요(RE_JO 확장). 이게 적재 태스크의 첫 검증점.
  - 평문 전처리는 `standards_prep`(표 언랩·헤더) 재사용.
- `build_simindex`/`export_dist`가 REG 자동 포함(doc_type 무관 전량 대상 — 기존 로직).
  export current 필터는 이미 `doc_type` 비TERMS를 포함하도록 보정됨(Phase 2) — REG도 포함되는지 확인.

## 5. 엔진 (변경 없음)

`db_similar(doc_type='REG')`로 규정 조회. Phase 2에서 추가한 doc_type 필터 그대로. 제목가중(랭킹개선)이
규정 제목 매칭에 그대로 작동. 코어 알고리즘·시그니처 불변.

## 6. UI — 3섹션

세 진입점(`/similar`, `/similar_text`, `/review`) 핸들러가 `db_similar`를 **세 번** 호출:
1. STANDARD(표준약관) top-3
2. REG(감독규정·법령) top-3
3. TERMS(타사) top-N (자사 제외)

렌더(`render_two_sections`를 `render_sections(list of (label, rows), query)`로 일반화):
```
📋 표준약관 대응 조문        (규범: 직접 벤치마크)
📖 관련 감독규정·법령        (규범: 최상위 근거)
🏢 타사 유사 조문            (관행)
```
- 규정 섹션도 발행처명·version_label(시행일) 노출. 각 규정 조문에 겹친표현 하이라이트 동일.
- 빈 섹션은 "관련 감독규정·법령 없음"으로 항상 표시.

## 7. 매핑 품질 게이트 (핵심 검증)

규정 적재 후, 샘플 약관 조문으로 `db_similar(doc_type='REG')` 결과를 **사람이 실측 평가**:
- 예: "해약환급금" 약관 조문 → 감독규정 해약환급금 조문이 상위? "보험금 지급" → 지급 관련 규정?
  "배당" → 배당 규정? "청약철회" → 청약철회 규정?
- **판정**: 5~8개 대표 주제에서 관련 규정이 top-3에 들면 similarity-only로 충분(주제사전 불필요).
  절반 이상 빗나가면 **주제사전 보강**(주제→규정 조문 큐레이션 수십 항목)을 후속 태스크로.
- 골든셋 확장: 통과한 주제 몇 개를 회귀 테스트로 고정.

## 8. 리스크·미결

- 규정 조문번호 `제N-M조` 파싱 — 적재 첫 검증점(clause_split 정규식 확장).
- 매핑 품질 불확실 → §7 게이트로 조기 판정, 약하면 주제사전(별도 태스크).
- 3섹션이 결과를 길게 만듦 → 각 섹션 top-3로 제한, 규정은 특히 압축.
- 법령 전문이 커 REG 조문이 많아짐 → doc_type 필터로 격리, 인덱스 크기 영향 점검.
- 규정 version_label(시행일) 정본 확인 — vault 파일 시행일 표기 사용, 최신성 사람 확인.

## 9. 후속 (Phase 4+)
- 인용 링크 해소(명시 인용 법령 펼치기).
- 주제사전 확장·유지보수 UI.
- 표준약관↔감독규정 근거 구조 매핑(표준약관=감독규정 별표15라 구조적 근거 존재).
