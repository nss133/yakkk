# 표준약관 브릿지 — 초안→표준약관→감독규정 매핑 설계

> 작성: 2026-07-13 · 상태: 설계 승인(대화)
> 대상: 감독규정 매핑 정밀도(현 2/6). VISION.md 개선 우선순위 ①.

## 1. 문제 (실측 근거: .superpowers/sdd/reg-mapping-eval.md)

초안→감독규정 직접 유사도는 6주제 중 2주제만 성공. 실패 유형이 둘로 갈림:
- **(a) 코퍼스 공백 3건**(보험금 지급 통상의무·청약철회·계약전 알릴의무): REG 4발행처에
  정면 대응 조문 자체가 없음(상법 소관). 지금은 무관 조문이 그럴듯하게 노출됨 — 오판 유도.
- **(b) 랭킹 미스 1건**(배당): 정답 제6-14조가 코퍼스에 있는데 인접조문에 밀림.

초안→표준약관은 강한 유사도로 잘 잡히므로(diff 기능으로 검증됨), 표준약관을 경유하는
**초안→표준약관(유사도)→감독규정(사전 매핑)** 브릿지로 우회한다.

## 2. 확정된 요구사항 (사용자 결정)

- **매핑 생성**: 빌드 시 자동생성 + 골든 JSON 수동보정.
- **섹션 구성**: 브릿지 결과 우선 표시(출처 라벨) + 직접 유사도 보충(중복 제거).
  골든에 "대응 없음"으로 선언된 조문은 오매핑 대신 정직한 공백 안내문 출력.

## 3. 설계

### 3.1 매핑 테이블 + 빌드 스크립트

- 새 테이블 `std_reg_map(std_clause_id INTEGER, reg_clause_id INTEGER, score REAL,
  source TEXT, note TEXT)` — source ∈ {'auto','golden','none'}.
  source='none' 행은 reg_clause_id NULL + note에 사유("상법 §651 소관" 등).
- 새 스크립트 `src/build_std_reg_map.py`:
  1. STANDARD 조문 172건 각각을 질의로 `db_similar(doc_type='REG', top_n=3)` 실행
  2. `BRIDGE_MIN_SCORE = 0.30` 이상만 자동 채택(source='auto', 조문당 최대 3행)
  3. `catalog/std_reg_golden.json` 로드해 오버라이드:
     - `{"std": "생보 제N조", "reg": ["감독규정 제6-14조", ...]}` → 해당 std 조문의
       auto 행 삭제 후 golden 행 삽입(score=1.0)
     - `{"std": "생보 제N조", "none": "상법 §651 소관(고지의무)"}` → source='none' 행 1개
     - 골든 키는 조문 식별자(발행처 member_cd + clause_no)로 지정 — clause_id는
       재적재마다 변하므로 golden에 쓰지 않음. 빌드 시 clause_id로 해석.
  4. 멱등: DELETE 후 재삽입. 미해석 골든 키(오타 등)는 경고 출력 + 빌드 실패(exit 1).
- 초기 골든 수록(실측 6주제 기반):
  - 배당 관련 표준약관 조문 → 보험업감독규정 제6-14조(계약자배당금의 산출 및 적립) 강제
  - 계약전 알릴의무 조문 → none("상법 §651 소관 — REG 정면 조문 없음")
  - 청약철회 조문 → none("금소법 §46(청약의 철회) 소관 — REG 4발행처 정면 조문 없음".
    구현 시 코퍼스에서 부재 재확인 후 확정, 발견되면 golden 강제로 전환)
  - 보험금 지급 조문 → none("상법 §658 소관")
  - ※ 구현 시 각 표준약관 조문번호를 실제 DB에서 확인해 골든 작성(스펙은 주제만 지정).

### 3.2 파이프라인 편입

- `update_all.py`: 4b(유사인덱스) 뒤에 `4c. 표준약관브릿지 build_std_reg_map.py` 추가
  (db_similar가 ngram_idf 필요 → 4b 이후 필수).
- `export_dist.py`: 테이블 튜플에 `std_reg_map` 추가. `--current-only`일 때는
  dist.clauses에 양쪽 clause_id가 존재하는 행만 복사(product_doc_map 패턴).
  STANDARD·REG 문서는 current-only에도 항상 포함되므로 실질 전량 복사됨.

### 3.3 런타임 (`_similar_blocks` 감독규정 섹션만 변경)

- 초안→표준약관 top-1 매치(diff와 동일한 저유사도 가드 0.25 공유)가 있으면:
  1. `std_reg_map`에서 top-1 표준약관 clause_id의 행 조회
  2. source='none'이면: "이 주제는 감독규정 정면 대응 조문 없음 — {note}" 안내문
  3. 매핑 행 있으면: REG 조문들을 먼저 표시, 각 행에
     "📎 표준약관 {clause_no}({title}) 경유" 출처 라벨(+golden이면 "검수됨" 표시)
  4. 기존 직접 유사도 REG 결과는 reg_clause_id 중복 제거 후 이어서 표시
     (라벨 "직접 유사")
- 표준약관 매치가 없거나 가드 미달이면: 현행 그대로 직접 유사도만.
- 매핑 테이블이 없는 구버전 DB로 실행 시: 조용히 현행 동작 폴백(반입 DB 하위호환).
- 세 화면(/similar·/similar_text·/review) 자동 적용. 런타임 stdlib 유지(테이블 조회만).

### 3.4 무변경 확인

`db_similar` 코어, 표준약관 diff 섹션, 타사 섹션, 기존 스키마 테이블 전부 불변.

## 4. 검증

- **단위**: build_std_reg_map — auto 채택 임계값, golden 오버라이드(강제/none), 멱등,
  미해석 골든 키 실패. 런타임 — none 안내문, 브릿지+직접 병합 중복 제거, 테이블 부재 폴백.
- **품질 게이트(6주제 재실측)**: 해약환급금·특별계정 PASS 유지 / 배당 골든 경유 PASS /
  공백 3건 오매핑 노출 → "없음" 안내 전환. **오답 표시 0**이 목표 지표.
  결과는 `.superpowers/sdd/reg-mapping-eval-v2.md`로 기록.
- 골든셋 테스트 고정(`tests/test_std_reg_bridge.py`) + 기존 스위트 무회귀.

## 5. 리스크

- 표준약관 top-1 오매치 시 브릿지도 오도 → 저유사도 가드 공유 + 출처 라벨로
  경유 경로를 투명하게 표시(심사자가 경유 조문 자체를 검증 가능).
- auto 매핑 임계 0.30은 초기값 — 게이트 실측으로 조정. 골든이 항상 우선하므로
  임계 민감도는 낮음.
- 골든 유지보수: 172조문 전수 아닌 필요분 점진 축적. 표준약관 개정 시 조문번호
  기반 키라 재적재에도 안정(조문번호 변경 시 빌드가 미해석 키로 실패해 드러남).

## 6. 후속 (범위 외)

- REG 코퍼스에 상법·금소법 추가(공백 3건의 근본 해소) / 주제사전(Phase 4)
- 감독규정 개정 → 영향 표준약관·자사 조문 역검색(VISION 활용 비전)
