# 약관 조문 비교 심사 도구 — Phase 1 (타사 비교) 설계

> 작성: 2026-07-08 · 상태: 설계 승인 대기
> 프로젝트: `~/yakgwan-db` (생보사 약관 DB, 10개사·295상품·조항 102만·현행판 426MB)

## 1. 배경·목적

법무팀은 상품부서가 만든 **신상품·개정 약관 초안**을 주기적으로 대량 심사한다.
심사는 초안 조문이 **감독규정·표준약관·타사 관행**에 비추어 문제가 없는지를 조문 단위로 짚어
수정의견을 내는 일이다. 이미 구축된 타사 약관 DB를 심사의 "비교 재료"로 삼아,
초안 조문마다 **닮은 타사 조문을 자동으로 끌어와 나란히 대조**하는 도구를 만든다.

전제 제약(불변):
- **LLM 불가·폐쇄망 지향** — 생성모델·외부 API 없이 동작해야 함(장기 목표는 폐쇄망 반입).
- 유사도는 **결정론적·설명가능**해야 함(왜 닮았는지 근거 제시).

## 2. 사용자·워크플로우

- 사용자: 법무팀 약관검토 담당.
- 지배 워크플로우: 초안(신상품/개정) 수령 → 조문별로 타사 동종상품 조문과 대조 → 수정의견.
- 두 가지 사용 모드:
  - **일괄**: 초안 파일 전체를 넣고 조문별 비교 리포트를 받아 전수 심사.
  - **단건**: 검토 중 특정 조문을 붙여넣거나 DB 내 조문에서 "닮은 타사 조문"을 즉석 조회.

## 3. 범위

**Phase 1 (본 설계) — IN**
- 비교 대상: **타사 동종상품 조문**(이미 DB에 있음).
- 유사도 엔진(엔진 A: FTS 후보 → 문자 n-gram TF-IDF 코사인 재정렬).
- 진입점 A: `search_app.py` 확장 — 단건/붙여넣기 즉석 비교 + 검색조건 **초기화 버튼**(버그②).
- 진입점 B: **docx 초안 업로드 → 일괄 심사 리포트**.
- 빌드 파이프라인에 **조문 제목 파싱 개선**(버그① 해결) + 유사도 인덱스 생성 단계 추가.
- 구동 환경: **일반 로컬 PC**(venv 자유). 폐쇄망 2파일 반입 형태는 Phase 1 범위 밖(§10 참조).

**OUT (후속 단계)**
- Phase 2: 표준약관(생보/질병상해/실손) 적재 후 비교 대상에 편입.
- Phase 3: 감독규정·법령 조문 연결(조문↔규정은 주제 매핑이라 유사도로 안 풀림 → 별도 접근).
- HWP/PDF 초안 파싱(초안 주력이 docx이므로 제외; 붙여넣기는 포맷 무관 항상 가능).

## 4. 아키텍처 개요

```
[빌드 시점 — 사외, 의존성 자유]                 [런타임 — 로컬 앱]
extract_index.py (제목파싱 개선)                 search_app.py (확장)
  → clauses(title 재추출)                          ├─ 검색 + 초기화 버튼
build_simindex.py (신규)                           ├─ /similar        (단건: DB 조문 → 유사 타사조문)
  → ngram_idf 테이블(캡 vocab)                      ├─ /similar_text   (붙여넣기 텍스트 → 유사조문)
export_dist.py → terms_dist_current.db             └─ /review         (docx 업로드 → 일괄 리포트)
                                                         │
                              simmatch.py (신규, stdlib) ─┘  ← 공용 유사도 엔진
                              docx_split.py (신규, python-docx) ← 진입점 B 전용
```

핵심 원칙: **무거운 계산은 빌드 시점, 런타임은 가볍게.**
- 빌드: 문자 n-gram IDF를 전 코퍼스에서 산출해 DB에 저장.
- 런타임: FTS로 후보를 수백 건으로 좁힌 뒤, 그 후보에 대해서만 stdlib 파이썬으로 코사인 계산.
  → 조문별 벡터를 저장하지 않아도 됨(후보 텍스트로 즉석 벡터화). 저장물 = IDF 테이블 하나.

## 5. 컴포넌트

### 5.1 유사도 엔진 (`simmatch.py`, stdlib 전용)

입력: 쿼리 조문 텍스트 + 필터(상품군, 자사 제외 여부 등). 출력: 유사 조문 top-N(회사별 그룹, 점수, 겹친 표현).

1. **정규화**: NFC → 공백·문장부호 제거 → 한글/영숫자만 → 문자 **3-gram + 4-gram** 집합.
2. **후보 생성(FTS/BM25)**: 쿼리 조문에서 변별력 있는 어절 상위 ~15개를 뽑아 `clauses_fts` OR 질의 →
   BM25 상위 ~300건. (같은 상품군·자사 제외 등 필터를 SQL WHERE로 병행.)
3. **재정렬(코사인)**: 후보 각각을 즉석 n-gram 벡터화(TF × IDF, L2정규화) → 쿼리와 코사인 →
   내림차순 top-N. **조문제목 일치 가산점**(쿼리 조문명과 후보 조문명이 같은 취지면 부스트).
4. **근거**: 상위 기여 n-gram/어절을 표시(왜 닮았는지). 결과 화면에서 겹친 표현 `<mark>` 강조.

IDF 저장(`ngram_idf` 테이블):
- df ≥ 2 이면서 df ≤ 0.5·N 인 n-gram만 유지(너무 희귀/너무 흔한 것 제외).
- 정보량 상위로 캡(목표 ≤ 300k행, ~10–20MB). 미등재 n-gram은 기본 고(高) IDF로 처리.

### 5.2 빌드 파이프라인 확장 (사외)

- **`extract_index.py` 제목 파싱 개선(버그①)**: 신한(92%)·NH(90%)·KB(86%)·흥국(82%)·메트(50%)·삼성(62%)
  조문 title 대량 공란 → 해당 사 PDF의 실제 "제N조 (제목)" 표기 레이아웃을 표본조사한 뒤
  헤더/제목 추출 규칙을 보강하고 재추출. (정확한 정규식은 구현계획에서 표본 확인 후 확정.)
- **`build_simindex.py`(신규)**: 현행판 코퍼스에서 n-gram IDF 산출 → `ngram_idf` 적재.
- `export_dist.py`가 `ngram_idf`를 반입 DB에 포함. `update_all.py` 체인에 두 단계 편입(멱등·증분).
- 조문분할 로직은 `extract_index`에서 `split_clauses(text)`로 분리해 docx 진입점과 공유(§5.4).

### 5.3 진입점 A — 단건/붙여넣기 (`search_app.py` 확장, stdlib)

- 조문 보기·검색결과 행에 **"닮은 타사 조문"** 버튼 → `/similar?id=<clause_id>`:
  해당 조문을 쿼리로 `simmatch` 호출, **자사(미래에셋) 제외·회사별 그룹·유사도%** 표시.
- **`/similar_text`**: 텍스트 붙여넣기 창(초안 조문 1~여러 개) → 각 문단을 조문분할 후 유사조문.
- **초기화 버튼(버그②)**: 검색폼에 "초기화" 추가 → 모든 필터·검색어 리셋(`/`로 이동).
- 기본 검색·조문열람 경로는 **stdlib만으로 계속 동작**(반입 2파일 모드 보존).

### 5.4 진입점 B — docx 업로드 → 일괄 리포트

- **`docx_split.py`(신규, `python-docx` 사용)**: docx 문단 추출 → `split_clauses()`로 조문 분할.
- `search_app.py`에 **`/review`** 라우트: multipart 업로드 수신 → 조문 분할 → 조문마다 `simmatch` →
  **표 리포트**(초안조문 | 최유사 타사조문 top-N | 유사도% | 조문 링크 | 겹친표현).
- **의존성 가드**: `python-docx` 미설치 환경에서는 업로드 기능만 비활성(검색·붙여넣기는 정상).
  → 반입 stdlib 경로와 로컬 리치 앱을 하나의 코드로 양립.
- multipart 파싱은 stdlib(`email.parser` 기반 소형 헬퍼)로 처리(3.13에서 `cgi` 제거 대응).

## 6. 데이터 모델 변경

- `ngram_idf(ngram TEXT PRIMARY KEY, idf REAL)` — 신규(반입 DB 포함).
- `clauses.title` — 값 채움 개선(스키마 변경 없음, 데이터 품질 향상).
- 그 외 스키마 변경 없음. 유사도 조회는 기존 `clauses`/`clauses_fts`/`documents`/`insurers` 재사용.

## 7. 데이터 흐름 (일괄 심사 예)

1. 심사자가 초안 `신상품약관.docx` 업로드.
2. `docx_split` → 조문 리스트 `[{clause_no, title, text}, …]`.
3. 조문마다 `simmatch(text, exclude_member='L34', prod_group=?)` → 유사 타사조문 top-N.
4. 리포트 렌더 → 심사자가 조문별로 타사 문구 대조하며 수정의견 작성.

## 8. 에러처리·엣지케이스

- **FTS 후보 0건**(희귀 문구): 어절 질의를 완화(상위 term 수 축소·부분일치) 후 재시도, 그래도 0이면
  "유사 조문 없음"으로 정직하게 표기(허위 매칭 금지).
- **초안 조문분할 실패**(제N조 헤더 부재): 문단 단위로 fallback 분할하고 경고 배지 표시.
- **너무 짧은 조문**(<30자, 목차/표지): 검색·비교 대상에서 제외(기존 30자 컷 재사용).
- **업로드 비 docx 파일**: 확장자·매직바이트 검증 후 거부 메시지.
- **동일 조문 다수 버전**(sha 다른 개정판): 회사·상품 단위로 접어 노출(중복 피로 감소).

## 9. 검증·테스트

- **골든셋**: 이미 아는 "동일 취지 조문쌍" 수십 개(예: 각사 '보험금 지급사유', '보험료 납입최고·해지')를
  수기로 라벨 → 초안측 조문을 쿼리로 넣었을 때 정답 타사조문이 **top-5 재현율**에 드는지 측정.
- **단위 테스트**: 정규화·n-gram·코사인(수작업 계산과 대조), `split_clauses`(경계 케이스),
  `docx_split`(샘플 docx).
- **성능**: 단건 질의 < 0.5s(후보 300 기준), 60조문 docx 일괄 < 30s 목표.
- **회귀**: 기존 검색·조문열람 경로가 stdlib만으로 여전히 동작하는지 확인.

## 10. 반입·의존성 원칙

- Phase 1 구동 = 일반 로컬 PC. 빌드 의존성(numpy 등)은 사외 자유, **런타임 유사도는 stdlib**.
- 로컬 리치 앱은 `python-docx`(+선택적 헬퍼)만 추가. `python-docx`는 순수 파이썬이라 폐쇄망 반입도 용이.
- 폐쇄망 반입 시: (a) 검색·붙여넣기 = 기존 "DB+search_app.py" 유지, (b) docx 업로드가 필요하면
  `python-docx`만 wheel로 동봉. 이 분리는 Phase 1 이후 별도로 확정.

## 11. 향후 단계

- **Phase 2 — 표준약관**: 생보/질병상해/실손 표준약관 수집·청킹 → `documents`에 `doc_type=STANDARD`로 적재
  → 같은 엔진으로 "초안 vs 표준약관" 비교. 엔진 변경 없음(재료만 추가).
- **Phase 3 — 감독규정·법령**: 조문↔규정은 주제 매핑 → 키워드/참조링크(korean-law·vault 연계) 방식 별도 설계.

## 12. 리스크·미결

- 제목 재추출 정규식이 회사별 레이아웃마다 달라 표본조사 필요(구현계획 1순위).
- IDF vocab 캡이 재현율을 해치지 않는지 골든셋으로 확인 필요.
- 상품군(prod_group) 필터가 초안-타사 매칭 정확도를 높이나, 초안의 상품군 자동판정은 불확실 →
  Phase 1은 상품군 필터를 **선택**으로 두고 전체 비교를 기본으로.
- (별개) 미커버 22건 수동확인·표준약관 확보는 Phase 2 착수 전 처리.
