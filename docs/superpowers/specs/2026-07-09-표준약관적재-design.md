# 표준약관 적재 — 준수성 심사 (Phase 2) 설계

> 작성: 2026-07-09 · 상태: 설계 승인 대기
> 선행: Phase 1(약관 조문 비교 심사, `v1.0-비교심사`). 엔진·검색앱은 그대로 두고 "재료(표준약관)"를 추가한다.

## 1. 배경·목적

Phase 1은 초안 조문을 **타사(업계 관행)** 조문과 비교한다. Phase 2는 여기에 **표준약관(규범 기준선)** 대조를 더해,
신상품·개정 약관 초안이 표준약관에 부합하는지(준수성)를 조문 단위로 확인하게 한다.
표준약관은 타사와 성격이 다른 "규범"이므로, 유사조문 결과에서 **항상 최상단에 별도로 강조**한다.

## 2. 범위

**IN**
- 표준약관 3종 적재: **생명보험 표준약관 · 질병·상해보험 표준약관(제3보험) · 실손의료보험 표준약관(5세대)**.
- 유사조문 결과(단건·붙여넣기·docx 일괄 심사) 최상단에 "표준약관 대응 조문" 섹션.
- `db_similar`에 `doc_type` 필터 추가(엔진 코어 알고리즘 불변).
- 표준약관별 **버전 라벨**(개정일/세대) 표기.

**OUT (후속)**
- Phase 3: 감독규정·법령 연결(조문↔규정 주제 매핑, 별도 설계).
- 표준약관 자동 최신화(고시 개정 추적) — 본 설계는 수동 소스 교체(매니페스트) 방식.
- 4세대 등 과거 세대 실손(현재 신상품 심사 목적엔 5세대가 규범).

## 3. 소스 — 로컬 우선, 매니페스트 방식

표준약관은 규범이므로 **버전이 곧 정확성**이다. vault 로컬 자료를 1차 소스로 하되,
표준약관별 소스를 **매니페스트(config)**로 분리해 최신본이 나오면 파일만 교체·재적재한다.

| 표준약관 | 소스(초기값) | 형식 | 비고 |
|---|---|---|---|
| 생명보험 표준약관 | vault: 생명보험 표준약관 전문(별표15 생명보험 섹션, 개정 2024.12.20) | markdown | 실손 외 현행 |
| 질병·상해보험 표준약관 | vault: `표준약관+질병상해보험(전문).md`(개정 2024.12.20) | markdown | 제3보험 |
| 실손의료보험 표준약관(5세대) | vault: `별표15_5세대_국내실손_표준약관_발췌_p203-312.pdf` | PDF | **신규 개정본** — 4세대 md 아님 |

- 매니페스트: `catalog/standards.json` — `[{key, name, version_label, member_cd, source_path, source_type(md|pdf)}]`.
- **정본 대조 의무**: 적재 전 각 소스의 개정일·세대를 확인하고, 고난도 쟁점 조문은 원소관(금감원 별표15 원문)과 대조. 편집본(vault 표) 특유의 표 래핑·발췌 누락 가능성을 점검. 확인 결과를 적재 리포트에 남긴다.
- ⚠️ 생명보험 표준약관의 정확한 vault 소스 파일(합본 별표15 vs 생보 추출본)은 적재 스크립트에서 "생명보험" 섹션만 취하도록 하고, 결과 조문수를 사람이 sanity check.

## 4. 데이터 모델·적재

### 4.1 스키마 (기존 재사용 + 최소 추가)
- `documents.doc_type`에 값 `'STANDARD'` 추가(스키마 변경 없음 — 이미 TEXT 컬럼).
- `insurers`에 표준약관 발행처 3행 추가: `(member_cd, name)` = `('STD_L','생명보험 표준약관')`, `('STD_H','질병·상해 표준약관')`, `('STD_S','실손 표준약관(5세대)')`. → 기존 `insurers` 조인이 그대로 동작, 결과에 발행처명이 곧 라벨.
- `documents.version_label`에 개정일/세대를 넣어 버전 표기에 재사용(예: `'2024.12.20 개정'`, `'5세대'`).
- `documents.prod_group`은 표준약관 종류 태그로 활용 가능(선택).

### 4.2 적재 스크립트 `ingest_standards.py` (신규, 빌드 시점)
1. `catalog/standards.json` 로드.
2. 소스가 markdown이면: 파일 읽기 → **마크다운 표 래핑 제거**(줄 앞뒤 `|` 및 셀 구분 정리) → 헤더 마크(`######` 등) 제거 → 평문화.
3. 소스가 PDF이면: 기존 `extract_index.extract_pdf_text` 재사용.
4. 평문 → 기존 **`clause_split.split_clauses`**로 조문 분할(`제N조(제목)` 인라인 형식이라 기존 파서가 처리). 별표·표(장해분류표 등)는 조문 아님 → 30자 컷/별표 처리로 자연 배제.
5. `documents`(doc_type=STANDARD, member_cd=발행처, version_label=버전) + `clauses` 적재. 재실행 안전(기존 STANDARD 삭제 후 재적재 또는 sha 기반 스킵).
6. 적재 후 sanity: 표준약관별 조문 수 출력(예: 생보 ~40, 질병상해 ~30, 실손 Nn).

### 4.3 파이프라인 편입
- `build_simindex.py`: STANDARD 조문 포함해 IDF 재빌드(코드 변경 없음 — `clauses` 전체 대상).
- `export_dist.py`: STANDARD documents·clauses가 자동 포함(기존 복사 로직). FTS도 포함.
- `update_all.py`: `ingest_standards.py`를 텍스트 추출 다음 단계로 편입(멱등).

## 5. 엔진 변경 (최소)

`simmatch.db_similar`에 `doc_type` 파라미터 추가:
- `doc_type='STANDARD'` → 표준약관 조문만 후보/재정렬.
- `doc_type='TERMS'`(기본) → 기존 타사 조문(자사 제외)만.
- FTS 후보 SQL에 `AND d.doc_type = ?` 추가. 나머지(코사인+BM25 블렌딩·제목 가산점) 불변.
- `exclude_member`는 TERMS 모드에서만 의미(표준약관엔 자사 개념 없음).

## 6. UI — 표준약관 최상단 별도 강조

세 진입점(`/similar`, `/similar_text`, `/review`) 모두 핸들러가 `db_similar`를 **두 번** 호출:
1. `db_similar(query, doc_type='STANDARD', top_n=3)` → 표준약관 대응 조문
2. `db_similar(query, doc_type='TERMS', exclude_member='L34', top_n=N)` → 타사 유사 조문

렌더:
```
📋 표준약관 대응 조문
  [생명보험 표준약관(2024.12.20)] 제3조 보험금의 지급사유  · 유사도 87% · 겹친표현 강조
────────────────────────
🏢 타사 유사 조문
  [삼성생명] 제3조 …  · 82%
  [한화생명] 제3조 …  · 79%
```
- 표준약관 섹션은 **항상 표시**(0건이면 "표준약관 대응 조문 없음"). 발행처명·version_label을 함께 노출해 어느 규범 버전인지 명시.
- `render_similar`는 그대로 재사용(표준약관도 insurers 조인으로 발행처명이 채워짐). 섹션 제목만 핸들러에서 구분.

## 7. 검증·테스트

- **골든셋 확장**(`tests/test_golden_recall.py`): 초안 조문 → 해당 **표준약관** 조문이 STANDARD 섹션 top-1에 오는지.
  - 예: "피보험자가 사망한 경우 보험금을 지급" → 생명보험 표준약관 "제3조 보험금의 지급사유".
  - "청약을 한 계약자는 … 청약을 철회" → 표준약관 청약철회 조문.
- **적재 단위 테스트**: 마크다운 표 래핑 제거(`_strip_md_table`) 유닛테스트; 소형 표준약관 샘플 → 조문 분할 검증.
- **sanity**: 표준약관 3종 조문 수가 상식적 범위인지, version_label이 채워졌는지.
- 회귀: 기존 TERMS(타사) 검색·유사비교가 그대로 동작(doc_type 기본값 TERMS).

## 8. 리스크·미결

- **정본 버전**: vault 편집본이 최신 고시와 다를 수 있음 → 적재 시 개정일 확인 + 고난도 조문 원문 대조. version_label로 사용자에게 명시(도구가 "어느 버전 기준"인지 투명).
- 생명보험 표준약관 소스 파일 확정(합본에서 생보 섹션 추출 vs 별도 추출본) — 적재 스크립트에서 섹션 경계 처리 + 조문수 sanity.
- 마크다운 표 래핑 제거가 조문 텍스트를 훼손하지 않는지(특히 `|`가 본문에 있는 경우) — 유닛테스트로 방어.
- 실손 PDF가 "발췌(p203-312)"라 전체 실손 표준약관이 아닐 수 있음 → 적재 리포트에 커버 범위 명시, 필요 시 전체본 확보.
- 표준약관은 insurer 없음 → `db_similar`의 `exclude_member`가 STANDARD 모드에서 무시되는지 확인(발행처 STD_* 는 자사 아님).

## 9. 후속 (Phase 3+)
- 감독규정·법령 조문 연결(주제 매핑 — 유사도 아닌 키워드/참조).
- 표준약관 세대별(4세대 등) 병행 조회(보유계약 심사용) — 필요 시.
